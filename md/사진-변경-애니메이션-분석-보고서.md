# 사진 변경 애니메이션 로직 분석 보고서

## 1. 개요

AI 가상 피팅 웹 서비스에서 "원본 상품 보기" 버튼 클릭 시 피팅 이미지에서 원본 이미지로 전환되는 애니메이션 로직을 분석합니다.

## 2. 관련 파일 및 컴포넌트

### 2.1 주요 파일
- `src/pages/Home.tsx` - 메인 페이지, 피팅 데이터 관리
- `src/components/common/ProductCard.tsx` - 개별 상품 카드, 애니메이션 구현
- `src/contexts/FittingContext.tsx` - 피팅 상태 관리
- `src/hooks/useProducts.ts` - 피팅 이미지 데이터 관리

## 3. 데이터 흐름 분석

### 3.1 전체 데이터 흐름
```
FittingContext → Home.tsx → useAllProductsFittingImages → ProductCard.tsx
     ↓              ↓                    ↓                      ↓
피팅 상태 관리   버튼 클릭 처리      피팅 데이터 조회        애니메이션 실행
```

### 3.2 상태 전파 과정

#### 1단계: 버튼 클릭 (Home.tsx)
```typescript
// 원본 상품 보기 버튼 클릭
onClick={() => {
  flushSync(() => {
    setIsAnimatingToOriginal(true);
  });
  setShowFitting(false);
  setTimeout(() => {
    setIsAnimatingToOriginal(false);
    resetFittingResults();
  }, 800);
}}
```

#### 2단계: 피팅 데이터 유지 (Home.tsx)
```typescript
const enableFittingResults = (showFitting && currentUserImageId !== null) || isAnimatingToOriginal;
const { fittingResults } = useAllProductsFittingImages(
  products?.products?.map((p) => p.product_id) || [],
  currentUserImageId,
  enableFittingResults
);
```

#### 3단계: ProductCard로 데이터 전달 (Home.tsx)
```typescript
<ProductCard
  product={{
    ...product,
    fittingImage: fittingResults[product.product_id]?.fitting_image || null,
  }}
/>
```

#### 4단계: 애니메이션 실행 (ProductCard.tsx)
```typescript
// 이미지 URL 계산
const foregroundImageUrl = isAnimating
  ? animationDirection === "left-to-right"
    ? product.image
    : previousShowFitting
    ? product.fittingImage || product.image
    : product.image
  : getDisplayImage();
```

## 4. 주요 상태 및 함수 분석

### 4.1 Home.tsx 상태

| 상태명 | 타입 | 역할 | 문제점 |
|--------|------|------|--------|
| `showFitting` | `boolean` | 피팅 모드 활성화 여부 | 버튼 클릭 시 즉시 `false`로 변경 |
| `isAnimatingToOriginal` | `boolean` | 원본보기 애니메이션 진행 상태 | flushSync로도 timing 문제 발생 |
| `currentUserImageId` | `number \| null` | 현재 사용자 이미지 ID | - |
| `fittingResults` | `Record<number, ProductFittingImageResponse>` | 피팅 결과 데이터 | enabled 조건에 따라 빈 객체가 됨 |

### 4.2 ProductCard.tsx 상태

| 상태명 | 타입 | 역할 | 문제점 |
|--------|------|------|--------|
| `isAnimating` | `boolean` | 애니메이션 진행 여부 | 다른 상태들과 동기화 문제 |
| `animationDirection` | `"left-to-right" \| "right-to-left"` | 애니메이션 방향 | - |
| `previousShowFitting` | `boolean \| null` | 이전 피팅 상태 | 설정 로직이 복잡하고 불안정 |

### 4.3 주요 함수

#### useAllProductsFittingImages (useProducts.ts)
```typescript
// enabled 조건이 false가 되면 즉시 빈 객체 반환
const useAllProductsFittingImages = (productIds, userImageId, enabled) => {
  // enabled가 false면 fittingResults = {}
  if (!enabled || !userImageId || productIds.length === 0) {
    return { fittingResults: {} };
  }
}
```

**문제점**: enabled가 false가 되는 순간 데이터가 사라짐

## 5. 핵심 문제 분석

### 5.1 State 업데이트 Timing 문제

```typescript
// 의도된 동작
setIsAnimatingToOriginal(true);  // 1. 먼저 애니메이션 상태 활성화
setShowFitting(false);           // 2. 그 다음 피팅 모드 해제

// 실제 동작 (React의 비동기 특성)
// 두 상태가 동시에 다음 렌더에 반영되어 순서 보장 안됨
```

**결과**: `useAllProductsFittingImages`의 enabled 조건이 잘못 계산됨

### 5.2 복잡한 상태 의존성 체인

```
Home.tsx 상태 변경 → useAllProductsFittingImages enabled 계산 → fittingResults 변경 → ProductCard props 변경 → 애니메이션 트리거 조건 계산
```

**문제점**: 체인의 어느 한 단계에서라도 문제가 발생하면 전체 동작 실패

### 5.3 ProductCard 애니메이션 조건 불만족

```typescript
// 애니메이션 트리거 조건 (4개 모두 true여야 함)
!isFittingLoading &&           // ✓ true
previousShowFitting !== null && // ❌ often false (null)
previousShowFitting !== showFitting && // ❌ often false
!isAnimating                   // ✓ true
```

**문제점**: `previousShowFitting` 설정 로직이 불안정

### 5.4 데이터 생명주기 불일치

```typescript
// 애니메이션 시작 전: fittingResults 존재
// 애니메이션 중: fittingResults 필요
// 800ms 후: fittingResults 삭제
```

하지만 실제로는 애니메이션이 시작되기 전에 `fittingResults`가 빈 객체가 되어버림

## 6. 문제 발생 시나리오

### 6.1 정상 시나리오 (의도)
1. 사용자가 "원본 상품 보기" 클릭
2. `isAnimatingToOriginal = true` 즉시 반영
3. `showFitting = false` 반영
4. `enableFittingResults = true` 유지 (애니메이션 중)
5. `fittingResults` 보존
6. ProductCard에서 애니메이션 트리거
7. 피팅 이미지 → 원본 이미지 전환
8. 800ms 후 정리

### 6.2 실제 시나리오 (문제)
1. 사용자가 "원본 상품 보기" 클릭
2. `isAnimatingToOriginal = true`, `showFitting = false` 동시 처리
3. React 렌더링에서 상태 순서 보장 안됨
4. `enableFittingResults = false` 계산됨
5. `fittingResults = {}` 빈 객체
6. ProductCard에서 `product.fittingImage = null`
7. 애니메이션 조건 불만족 또는 이미지 없음
8. 흰색 배경 또는 애니메이션 없음

## 7. 근본적인 설계 문제

### 7.1 과도한 상태 의존성
- 6개 이상의 상태가 복잡하게 연결됨
- 하나의 상태 변경이 전체 시스템에 영향

### 7.2 분산된 책임
- Home.tsx: 데이터 관리
- ProductCard.tsx: 애니메이션 로직
- useProducts.ts: 데이터 조회
- 각 컴포넌트가 다른 컴포넌트의 상태에 의존

### 7.3 Timing에 민감한 로직
- React의 비동기 상태 업데이트에 의존
- flushSync 같은 우회 방법 필요

## 9. 상세 기술 분석

### 9.1 React State 업데이트 메커니즘

React의 상태 업데이트는 기본적으로 **비동기적**이며 **배치 처리**됩니다:

```typescript
// 이 코드에서
setIsAnimatingToOriginal(true);
setShowFitting(false);

// React는 내부적으로 다음과 같이 처리:
// 1. 두 상태 업데이트를 큐에 추가
// 2. 다음 렌더 사이클에서 일괄 처리
// 3. 순서 보장 없음
```

### 9.2 flushSync의 한계

```typescript
flushSync(() => {
  setIsAnimatingToOriginal(true);
});
setShowFitting(false);
```

`flushSync`는 동기적 업데이트를 강제하지만:
- **성능 저하**: 리렌더링을 즉시 발생시킴
- **부작용**: 다른 컴포넌트의 상태에 영향
- **복잡성 증가**: 코드 이해도 저하

### 9.3 useEffect 의존성 배열 문제

```typescript
// ProductCard.tsx
useEffect(() => {
  // 복잡한 조건 로직
}, [showFitting, isFittingLoading, previousShowFitting, isAnimating, product.product_id]);
```

**문제점**:
- 의존성이 많아 예측 불가능한 재실행
- 무한 루프 가능성
- 디버깅 어려움

## 14. 결론 및 최종 권장사항

### 14.1 현재 상황 요약
현재 애니메이션 로직은 다음과 같은 심각한 문제들을 가지고 있습니다:

1. **복잡성**: 6개 이상의 상태가 얽혀있는 복잡한 구조
2. **불안정성**: React의 비동기 특성으로 인한 타이밍 문제
3. **유지보수성**: 하나의 수정이 전체 시스템에 영향
4. **테스트 어려움**: 상태 의존성으로 인한 테스트 불가능
5. **성능 문제**: 불필요한 리렌더링과 API 호출

### 14.2 최종 권장 해결책

#### Option A: 완전한 재설계 (권장)
```typescript
// 1. 단일 상태로 통합
type AnimationState = {
  mode: 'original' | 'fitting';
  isTransitioning: boolean;
  savedImages: {
    fitting: string | null;
    original: string;
  } | null;
};

// 2. 순수 함수로 로직 분리
const getDisplayImage = (state: AnimationState) => {
  if (state.isTransitioning && state.savedImages) {
    return state.mode === 'original' 
      ? state.savedImages.original 
      : state.savedImages.fitting;
  }
  return state.mode === 'fitting' ? state.savedImages?.fitting : state.savedImages?.original;
};

// 3. 단순한 전환 로직
const handleModeChange = () => {
  setState(prev => ({
    ...prev,
    isTransitioning: true,
    savedImages: {
      fitting: currentFittingImage,
      original: currentOriginalImage
    }
  }));
  
  setTimeout(() => {
    setState(prev => ({
      mode: prev.mode === 'original' ? 'fitting' : 'original',
      isTransitioning: false,
      savedImages: null
    }));
  }, 800);
};
```

#### Option B: 점진적 개선
1. **1단계**: 현재 코드에서 디버깅 로그 제거 (완료)
2. **2단계**: `flushSync` 대신 `useLayoutEffect` 사용
3. **3단계**: 상태 수 줄이기 (`isAnimatingToOriginal` 제거)
4. **4단계**: 순수 함수로 이미지 URL 계산 로직 분리

#### Option C: 애니메이션 제거 (임시 해결책)
```typescript
// 애니메이션 없이 즉시 전환
const handleOriginalView = () => {
  setShowFitting(false);
  // 애니메이션 관련 로직 모두 제거
};
```

### 14.3 구현 우선순위

1. **High Priority**: Option A (완전한 재설계) - 근본적 해결
2. **Medium Priority**: Option B (점진적 개선) - 안정성 확보
3. **Low Priority**: Option C (애니메이션 제거) - 임시 방편

### 14.4 마무리

현재 애니메이션 시스템은 **과도한 엔지니어링**의 전형적인 사례입니다. 단순한 이미지 전환 기능에 너무 많은 상태와 로직이 투입되어 오히려 불안정해졌습니다.

**핵심 교훈**:
- 복잡한 상태 관리보다는 단순한 구조가 더 안정적
- React의 특성을 이해하고 그에 맞는 패턴 사용
- 기능 구현 전 설계 단계에서 충분한 검토 필요

**즉시 조치 사항**:
1. 현재 불안정한 코드를 안정적인 버전으로 롤백
2. 새로운 설계로 다시 구현
3. 충분한 테스트 후 배포
